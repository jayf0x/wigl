# Backlog

Atomic tasks, not a priority-ordered roadmap. Each entry is a problem or gap
someone (or an agent) could pick up and finish in one sitting without asking
"what does this mean?" first. Product ideas live in `docs/future-ideas.md`;
this file is defects, gaps, and known ceilings in what already exists.

Sections describe what *kind* of change an entry is, not how urgent it is —
pick whichever section matches the task at hand, there's no priority order
within or across them:

- **Features** — capability that doesn't exist today.
- **Bugs** — existing functionality doesn't do what it's supposed to.
- **Improvements** — same behavior, better/cleaner/faster/safer underneath.

Rules for keeping this file real:

- Every entry must be actionable right now. If it's not, it doesn't belong here.
- When an item is fixed, delete it. Don't check it off and leave it, don't move it to a "done" or "changelog" list — git history is the changelog.
- When a decision is made and closed with no remaining follow-up work, delete it too. Don't keep a "Decided / won't fix" graveyard.
- No dates, no "as of this session", no "swept on", no references to when something was added or by whom. This file describes the current state of the codebase, not its history.
- If a fix is genuinely speculative (no concrete trigger yet), say so plainly and give the trigger condition — don't leave vague "might be nice" entries.
- Prefer deleting or rewriting a stale entry over leaving it half-true when the code has moved past it. New entries follow this same file's own rules — keep it that way.

## Features

- [ ] **Multiple instances of one widget type** — today a widget is exactly one folder = one window = one instance (`App.tsx`'s `import.meta.glob`, folder name is the id). `<Widget w h col row>` itself doesn't care about identity, so no widget's own code would need to change; the gap is discovery (folder name *is* the id) and storage (each instance needs its own key, not a shared one). Trigger: a concrete want for two instances of one widget type (e.g. two clocks on different timezones). Fix shape: needs an explicit place to declare "N instances of folder X, with these overrides" — a small, hand-edited config naming active instances, not a full render-tree registry.
- [ ] **Monitor add/remove without an app relaunch** — `screen-<i>` windows are created once in Rust's `setup()` ([lib.rs](src-tauri/src/lib.rs)) from `available_monitors()` at launch. Plugging in or removing a display doesn't add/destroy a window or migrate its widgets. Trigger: this becomes the normal complaint once someone actually docks/undocks a laptop with wigl running. Fix shape: listen for a monitor-change signal (polling `available_monitors()` is the only cross-platform option Tauri exposes today) and reconcile: spawn new screen windows, and for removed monitors, reassign their widgets' `m` to monitor 0 rather than losing them.
- [ ] **Escape cancels a drag** — deliberate gap, not an oversight: `screen-*` windows are `alwaysOnBottom` and rarely become key, so a `keydown` listener isn't a reliable place to catch Escape. Trigger: revisit if/when there's a reason one of these windows does become key (e.g. a command palette), since that would give you a reliable focus target to hang the listener off.
- [ ] **`<Widget>` visibility/hide prop** — no way to hide a widget without deleting its folder. Trigger: a concrete need for this arises. Fix shape: the grid self-report channel (`WidgetGridReport`/`reportGrid` in [Desktop.tsx](src/wigl/Desktop.tsx)) already lets `<Widget>` tell Desktop things about itself beyond size/position — add `hidden?: boolean` to that report and have Desktop skip rendering/hit-testing it. Small, contained, once needed.
- [ ] **Smoother `bun run tauri dev` iteration for widget add/edit** — adding or editing a widget currently wants a rebuild+relaunch to see reliably, when normal Vite HMR/Fast Refresh should get closer than that. Not yet diagnosed: needs someone to actually watch `tauri dev`'s behavior when adding a new widget folder vs. editing an existing widget file, and check whether `import.meta.glob`'s HMR handling of the now-lazy widget map (each entry is a `React.lazy()`-wrapped loader, see `App.tsx`) or the multi-webview-window setup (each `screen-<i>` window is its own HMR client) is where it breaks down, before designing a fix.
- [ ] **No secret/token storage primitive** — `useStorage` (sqlite kv blob, see `docs/widgets.md`) is the only persistence a widget has today; a widget needing an API key or OAuth token would land it there in plaintext, since nothing else exists. Trigger: the first widget that actually needs a token (weather API, calendar OAuth, etc). Fix shape: one small `#[tauri::command]` pair (`secrets_get`/`secrets_set`, keyed by a caller-supplied name) writing a single JSON file to the app's data dir, `chmod 600`, atomic tmp-file+rename — no OS-keychain crate, no cross-platform branching, just a permissions-restricted file. This is one of the few cases that clears wigl's "shell out unless truly impossible" bar: `chmod`-after-write via `sh -c` is doable, but atomic tmp+rename plus keeping the value out of shell argv/history genuinely wants a native command instead of a shell one-liner.

## Bugs

- [ ] **Cross-monitor drag col/row offset on screens of different sizes** — reported symptom: dragging a widget from a smaller screen to a bigger one (or vice versa) lands it at an unexpected position. `GridItem`/`SavedPositions`/the drag message payloads were renamed `x`/`y` → `col`/`row` for a clear API, but the underlying fix isn't implemented yet — a static read of `onPointerMove`'s foreign-monitor branch in [Desktop.tsx](src/wigl/Desktop.tsx) (target `col`/`row` come from `pxToCol`/`pxToRow` against the *target* monitor's own `colsForWidth`, which already looks size-aware) didn't turn up an obvious cause, so the repro needs pinning down first. Likeliest suspects to check while reproducing: `d.offX`/`d.offY` (captured once at drag-start from the source monitor, reused verbatim once the cursor is over a different monitor) and `TILING.padding.top` (a flat 48px menu-bar clearance applied to every monitor, including ones without a menu bar). Fix shape once confirmed: a small helper that recomputes a relative `col`/`row` for monitor B from monitor A's `col`/`row` and each monitor's own column/row capacity, instead of reusing a raw pixel offset across monitors.
- [ ] **Windowed-mode interactive flows unverified end-to-end** — the Wayland/GNOME windowed flow (`docs/architecture.md`) has been confirmed to build and launch cleanly on a real Linux/GNOME machine (correct mode detection, clean log, `tauri://localhost` page load), and click-through was tried live and reverted, confirmed not worth keeping without positioning/stacking control (see `docs/architecture.md`) — but drag/drop within the single window, `useStorage` read/write, the per-widget error boundary actually catching a crash, and the repos widget's Linux-specific shell paths (`src/widgets/repos/commands.ts`'s `xdg-open`/VS Code CLI/GitHub Desktop community-fork branches) still haven't been exercised interactively — no input-synthesis tool was available to script clicks/drags in the environment this was checked from. Trigger: someone with hands on a real Linux/GNOME machine spends 15–20 minutes actually dragging widgets, toggling repos-widget settings, and clicking its action buttons.
- [ ] **First-launch layout settles in report order, not globally** — when widgets mount with no saved position yet (true first launch, or a newly added widget), each starts at `TILING.defaultSize` and corrects to its real `w`/`h` via `reportGrid` ([Desktop.tsx](src/wigl/Desktop.tsx)) as its own `<Widget w h>` mounts and reports. `reflow()` runs once per incoming report, in whatever order those reports happen to arrive — not as one pass with every widget's true size known upfront. The settled layout for several never-before-seen widgets can therefore depend on mount/report order instead of being deterministic. Not currently visible (3 widgets, no surprising sizes), but will show up as odd gaps/overlaps once more widgets ship without ever having been dragged into place. Trigger: adding enough new widgets at once to notice.

- [ ] **(Low priority) Drag ghosting/residue in windowed mode (GNOME/Wayland)** — dragging a widget leaves a visual trail of its previous position(s) that doesn't clear until something forces a repaint. Not reproduced interactively in this pass (no input-synthesis tool available in the environment this was investigated from — no `ydotool`/`wtype`/`xdotool` installed, and synthesizing real mouse events via `/dev/uinput` on someone's live session is too invasive to do unprompted), so this is a hypothesis, not a confirmed root cause. Likeliest mechanism: `.wigl-widget` (`src/App.css`) is `will-change: transform` and gets its position written via a raw `el.style.transform` on every `pointermove` during a drag ([Desktop.tsx](src/wigl/Desktop.tsx)'s `onPointerMove`) — a GPU-layer-promoted transform animated at pointer rate is exactly the pattern that trips known WebKitGTK compositor damage-region bugs, and `lib.rs`'s `run()` already sets `WEBKIT_DISABLE_DMABUF_RENDERER=1` by default on Linux (worked around a *different* bug — blank windows on some GPU/external-display combos — see `docs/debugging.md`), which forces a fallback rendering path that's a plausible place for a *new* repaint/damage bug to show up. Cheap things to try first, in order: (1) A/B test with DMA-BUF re-enabled — launch with `WEBKIT_DISABLE_DMABUF_RENDERER= bun run qa:app` (empty value overrides the default) and see if ghosting disappears; if so the two workarounds are in tension and need a driver-specific condition instead of a blanket default. (2) `WEBKIT_DISABLE_COMPOSITING_MODE=1` (already noted in `docs/debugging.md` as a blunter fallback) as a second data point. (3) If neither clears it, it's likely a Mesa/driver-specific WebKitGTK compositor bug rather than anything in this app's CSS/drag code — check `journalctl --user -b 0 | grep -i webkit` for driver-level messages while reproducing. Trigger to pick this up: someone with a GNOME/Wayland machine and a spare 20 minutes to actually drag a widget and try the two env-var A/B tests above.
- [ ] **(Very low priority) GNOME-native overlay flow via a Shell extension** — Wayland/GNOME sessions currently get the windowed flow (`docs/architecture.md`) instead of the real desktop-overlay one, because stock GNOME Wayland refuses a client absolute positioning and always-below/always-on-top stacking (click-through alone isn't the gap — that part actually works over Wayland, it just isn't useful without one of those two to keep the window out of the way; see `docs/architecture.md`). A GNOME Shell extension ([desktop-widgets](https://github.com/NiffirgkcaJ/desktop-widgets)-style) could plausibly grant enough positioning/stacking control to run the real per-monitor overlay flow under GNOME too. Not worth pursuing casually: it's an extra install step for the user (extensions aren't bundled with the app), version-fragile against GNOME Shell updates in a way the current windowed flow isn't, and would need real QA on the positioning/stacking contract before it's safe to switch GNOME users onto it by default. Trigger: someone's actually bothered enough by the windowed flow's remaining UX gap (window doesn't stay pinned below/above other apps, doesn't span monitors) to want to try it.

## Improvements

- [ ] **Poll-based storage sync** — every key in every window still spawns a `sqlite3` process every 3 s to detect *external* changes (another widget, a CLI script). `widget_layout` now also broadcasts over `emit`/`listen` (`wigl-layout` in [Desktop.tsx](src/wigl/Desktop.tsx)) so cross-monitor drags don't wait on the poll, but that's one key, not a general fix — every other key (`calendar_events`, `repos_sort_by`, ...) still only updates on the next poll tick. Trigger: visible flicker, ~6+ widgets, or another key needing the same near-instant cross-window sync.
- [ ] **Grid reflow only compacts vertically** — `reflow()` in [grid/math.ts](src/wigl/grid/math.ts) pushes collisions straight down and gravity-compacts upward, with no horizontal repacking. A monitor that's had a lot of widgets dragged around can end up as one tall column with wasted width beside it, even though a compact arrangement exists. Not wrong, just not optimal. Trigger: this is visibly annoying in daily use — the fix (occupy the first horizontal gap, not just push down) is a moderate rewrite of `reflow`, not a small patch, so don't do it speculatively.
- [ ] **`useQuery`'s SQL cache never evicts** — `src/wigl/storage/useQuery.ts`'s `useSql: true` path writes `query_<key>` rows into the shared kv table but nothing ever deletes one, so a renamed or retired query key leaves a dead row behind forever (harmless, just clutter). Trigger: enough renamed/retired query keys to notice in the DB. Fix shape: none needed until then — a manual `DELETE FROM kv WHERE key LIKE 'query_%'` sweep is enough when it does.
- [ ] **Anchor field: SVG rewrite, not Canvas** — the drag-time "anchor field" (cross marks on every grid cell corner, brightening near the cursor, locking onto the drop target's four corners) in [Desktop.tsx](src/wigl/Desktop.tsx) is drawn on a `<canvas>` with `CanvasRenderingContext2D`, redrawn every frame via `requestAnimationFrame` (`drawField`/`wakeField`, plus the `canvas`/`cursor`/`ghostCell`/`field` refs). Planned rework, not a repair — the canvas version works fine. Why: per-frame full-canvas redraws for a mostly-static grid of crosses is more machinery than the effect needs; SVG lets individual cross elements be targeted/animated with CSS instead of hand-rolled canvas math, and composes better with the rest of the DOM. Fix shape: one `<svg>` (or a grid of small SVG `<path>`/`<line>` elements) positioned like today's `.wigl-field` (`src/App.css`), reusing `colToPx`/`rowToPx`/`spanToPx` ([grid/math.ts](src/wigl/grid/math.ts)) for cross placement instead of re-deriving pixel math. Preserve: cursor-proximity brightening, drop-target corner lock-in, `prefers-reduced-motion`, and the three `TILING.field.show` modes (`src/wigl/grid/config.ts`). `ACCENT` (top of `Desktop.tsx`) must stay in sync with the `--wigl-accent` CSS variable regardless of implementation. Delete the canvas code (`drawField`, `wakeField`, the canvas ref/element, `field` ref) once the SVG version is in and verified, rather than keeping both paths around.
